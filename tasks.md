# Production MVP Tasks

Related docs: [README](README.md), [Architecture](docs/architecture.md), [User Journey Scenarios](docs/user-journeys.md), [Agent Notes](AGENTS.md), [Claude Notes](CLAUDE.md).

| # | Priority | Problem | Solution | Acceptance | Impact |
|---:|---|---|---|---|---|
| 1 | P0 | Public invite submission is handled as direct browser writes to Supabase, so token validation, expiry, one-time consumption, date validation, moderation, CAPTCHA, and anti-abuse checks are not enforced in one trusted place. | Move invite submission to a server-side endpoint, preferably a Supabase Edge Function such as `submit-invite`, using the service role key only inside the function. | Public invite flow creates exactly one invite for a valid slot/date/link, rejects expired or used links, consumes one-time links atomically, requires CAPTCHA for public profile submits, and never exposes service credentials to the browser. | Makes public submission safe enough to run outside Lovable and prevents spam, invalid rows, and token bypasses. |
| 2 | P0 | Invite link types and RLS policies do not match the app. The app uses `3_day` and `7_day`, the schema allows `exp_3d` and `exp_7d`, and public-link policies check `type = 'public'`. | Standardize link type values across migrations, generated types, app code, and policies. Prefer the spec names `one_time`, `exp_3d`, and `exp_7d`, then update UI labels separately. | Link creation succeeds, link lookup works anonymously only for valid unexpired links, one-time links are refreshed after successful submission, and regression tests cover all link types. | Fixes a core flow that can otherwise fail before hosting even matters. |
| 3 | P0 | Accepting an invite is not transactional or idempotent. The client updates invite status first, then inserts a date, and no safety pack or SMS is guaranteed. | Implement an `accept-invite` Edge Function or Postgres RPC that validates ownership, locks the invite, updates status, creates the date, creates the safety pack draft, logs notification work, and sends or queues SMS. | Double-clicking Accept or retrying the request produces one accepted invite, one date, one safety pack draft, and one notification/log entry. Partial failure is either rolled back or marked for retry. | Prevents duplicate dates and broken accepted invites, which is essential for user trust. |
| 4 | P0 | Safety Pack activation only updates database status from the browser. There is no scheduled check-in SMS, signed acknowledgement endpoint, or escalation job. | Add functions for `activate-safety-pack`, `ack-safety-pack`, and an escalation/check-in worker using Supabase scheduled functions, cron, or a budget-friendly external scheduler. | Activating a pack schedules a check-in; OK marks completed; Call/Emergency immediately alerts trusted contacts; missed check-in after grace period triggers escalation once. | Turns the Safety Pack from UI into a real safety workflow. |
| 5 | P0 | SMS automation is specified but not implemented: no provider integration, delivery retry handling, or real notification pipeline. | Choose an SMS provider, add provider secrets to Supabase, create a small messaging service inside functions, and use `notification_log` for idempotency, status, and retries. | Accept notifications, host digests, check-ins, and escalations can be sent in a staging project; failures are logged with provider IDs/statuses and can be retried safely. | Enables the product's highest-value automation. |
| 6 | P1 | Auth is email/password, while the spec calls for phone OTP sign-in/up and phone verification for invitees when required. | Decide whether to use Supabase phone auth or keep email temporarily. If phone OTP remains in scope, refactor host auth and invitee verification around phone OTP. | Host login matches the selected auth strategy; invitee phone-required submissions cannot complete without verified phone; redirect URLs work on localhost, staging, and production. | Aligns identity and SMS workflows with the MVP promise. |
| 7 | P1 | Turnstile CAPTCHA is required for public profile submits but is not implemented. | Add Turnstile to the public invite wizard for `/:handle` and verify the token server-side in the submit function. | Public profile submissions without a valid Turnstile token are rejected server-side; token links can follow the chosen policy. | Reduces spam on public profiles. |
| 8 | P1 | Auto-decline rules and keyword moderation are configured in the UI/schema but not reliably enforced on submit. | Enforce enabled questions, auto-decline rules, note length, and keyword moderation in the server-side submit function. | Unsupported or missing required answers are rejected; configured auto-decline rules set status appropriately; flagged content shows moderation status to the host. | Keeps screening behavior consistent and prevents client-side bypass. |
| 9 | P1 | Safety link tokens are generated in the browser and acknowledgement links are not exposed as trusted server endpoints. | Generate signed or high-entropy safety tokens server-side, store only what is needed, and route acknowledgement URLs to a public Edge Function. | Tokens cannot be guessed, are scoped to one pack/action, and acknowledgement writes are idempotent. | Protects sensitive safety actions from spoofing. |
| 10 | P1 | Public schedule data is assembled by multiple browser queries and depends heavily on permissive RLS. | Add a `public-schedule` Edge Function or carefully audited RPC that returns only the public schedule payload needed by the UI. | Anonymous users can fetch valid public/link schedules but cannot enumerate private schedules, invitee data, or host private fields. | Reduces privacy risk and simplifies RLS. |
| 11 | P1 | Hosting-specific SPA fallback config is missing. Deep links such as `/:handle`, `/i/:token`, and `/dates/:id` may 404 on static hosts. | Add one deployment config for the chosen host: `vercel.json`, `netlify.toml` or `public/_redirects`, or Cloudflare Pages routing config. | Reloading every app route works in staging and production. | Required for independent static hosting polish. |
| 12 | P1 | Environment handling is not production-ready. `.env` is tracked and there is no `.env.example` documenting required variables. | Untrack real secrets, add `.env.example`, document frontend env vars and function secrets separately, and rotate exposed credentials if needed. | Fresh clone can be configured from `.env.example`; no secret-bearing `.env` file is committed; browser only receives publishable keys. | Prevents credential leakage and makes deployment repeatable. |
| 13 | P2 | Supabase functions are only partially present in the repo; the first Telegram webhook exists, but the broader trusted backend is still not versioned. | Keep every Edge Function in `supabase/functions`, add local tests, and deploy them from CLI/CI instead of dashboard-only edits. | All production backend code is committed beside migrations; functions can be tested locally and deployed repeatably from CLI/CI. | Makes the product independent from Lovable/dashboard-only state. |
| 14 | P2 | There is no local/staging verification workflow outside Lovable. | Add a README section or checklist for `npm ci`, `npm run build`, `npm run lint`, Supabase local/linked setup, function serving, and staging smoke tests. | A developer can run the app locally against staging Supabase and complete landing, signup, schedule setup, public invite, accept, and safety-pack flows. | Reduces deployment risk and makes future changes safer. |
| 15 | P2 | Demo pages exist but the sandbox behavior is not clearly isolated from production invite tables. | Keep demo submissions fully client-only, or route them to a dedicated demo table/function with purge policy. | Demo invite flow cannot create real invites, dates, SMS sends, or safety events. | Preserves trust and avoids noisy production data. |
| 16 | P0 | The next invite flow requires SMS phone verification, but the current public invite flow accepts phone numbers as unverified text and writes directly from the browser. | Implement the end-to-end web invite flow through `send-phone-otp`, `verify-phone-otp`, and `submit-invite`, with SMS provider integration for UAE, Turkey, and Singapore. | Visitor can submit an invite in the web app without Telegram only after phone OTP succeeds; invalid/expired OTP fails; one active pending invite per verified phone per host is enforced; host receives exactly one pending invite. | Establishes trusted visitor identity and anti-duplicate behavior while preserving the web-first invite experience. |
| 17 | P0 | Hosts need Telegram invite administration, but Telegram is not linked to host accounts and invite decisions are still client-side. | Implement `telegram-webhook`, host Telegram linking, new-invite notifications, and inline Accept/Decline buttons backed by server-side `accept-invite` and `decline-invite`. | Linked host receives a Telegram message for a new invite, can accept or decline from Telegram, and web dashboard/date state updates consistently with no duplicate dates. | Gives hosts the intended admin channel and moves invite lifecycle to trusted backend logic. |
| 18 | P0 | Visitors should learn about accepted invites through Telegram after opting in, but invite submission should not require Telegram. | Keep the implemented post-submit CTA, invite-specific Telegram deep links, and visitor Telegram linking; add accepted-invite notifications with host-approved contact details. | Visitor can submit via web first, then optionally link Telegram; if linked, acceptance sends a Telegram notification; if not linked, no Telegram notification is attempted; host phone is never shared by default. | Adds accepted-invite notification without blocking web submission or forcing Telegram too early. |
| 19 | P1 | Host profile visibility, discovery visibility, and accepted-contact sharing are not modeled separately. | Add host settings for public profile visibility defaulting on, discovery visibility defaulting on, Instagram handle, linked Telegram, and accepted-contact channel (`telegram` or `instagram`). | Turning public profile off hides `/:handle`; discovery only includes profiles with public profile on, schedule active, and discovery on; accepted notifications include only the selected contact method. | Separates privacy controls from discovery and makes contact sharing explicit. |
| 20 | P1 | Safety Pack still has no real reminder or emergency notification path. | Implement Safety Pack Telegram check-in reminders plus SMS trusted-contact alerts for missed check-in or manual Emergency action. | Activating Safety Pack schedules a Telegram check-in; All good prevents escalation; Emergency or missed check-in sends one SMS to trusted contacts and records delivery status. | Turns the safety promise into an operational workflow while keeping trusted contacts SMS-only for now. |
| 21 | P1 | Telegram discovery is a new product surface and currently has no location context, profile eligibility, skip/invite state, or web handoff. | Implement Telegram "View profiles nearby" browsing using first viewed/invited host city as initial context, then Telegram native location or manual city, with one-profile-at-a-time Invite/Skip actions. | Visitor can enter discovery from post-invite CTA or bot menu, browse eligible profiles one by one, skip profiles, and open the web invite flow for a selected profile. | Adds the discovery loop without replacing the richer web invite wizard. |
