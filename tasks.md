Active tasks below are open production work. Implemented slices are moved to [Completed / Closed Tasks](#completed--closed-tasks) so the active table stays usable.

## Active Tasks

| # | Priority | Problem | Solution | Acceptance | Impact |
|---:|---|---|---|---|---|
| 1 | P0 | Public invite submission is handled as direct browser writes to Supabase, so token validation, expiry, one-time consumption, date validation, moderation, CAPTCHA, phone verification, and anti-abuse checks are not enforced in one trusted place. | Move invite submission to a server-side `submit-invite` Supabase Edge Function using the service role key only inside the function. | Public invite flow creates exactly one invite for a valid slot/date/link, rejects expired or used links, consumes one-time links atomically, verifies CAPTCHA for public profile submits, requires verified phone when configured, and never exposes service credentials to the browser. | Makes public submission safe enough to run outside Lovable and prevents spam, invalid rows, and token bypasses. |
| 2 | P0 | Invite link types and RLS policies do not match the app. The app uses `3_day` and `7_day`, the schema allows `exp_3d` and `exp_7d`, and public-link policies check `type = 'public'`. | Standardize link type values across migrations, generated types, app code, and policies. Prefer the spec names `one_time`, `exp_3d`, and `exp_7d`, then update UI labels separately. | Link creation succeeds, link lookup works anonymously only for valid unexpired links, one-time links are refreshed after successful submission, and regression tests cover all link types. | Fixes a core flow that can otherwise fail before hosting even matters. |
| 3 | P0 | Accepting an invite has moved to `accept-invite`, but full transactional idempotency is not finished: there is no database lock/unique constraint for one date per invite and no Safety Pack draft creation guarantee. | Upgrade `accept-invite` to a Postgres RPC or add database uniqueness/locking, create the Safety Pack draft, and make notification logging idempotent. | Double-clicking Accept or retrying the request produces one accepted invite, one date, one safety pack draft, and one notification/log entry. Partial failure is either rolled back or marked for retry. | Prevents duplicate dates and broken accepted invites, which is essential for user trust. |
| 4 | P0 | Twilio Verify primitives exist, but the production SMS path is not fully activated and Safety Pack SMS still has no sender/status callback implementation. | Configure Twilio Function Secrets, deploy `send-phone-otp` and `verify-phone-otp`, enable `VITE_PHONE_VERIFICATION_MODE=twilio` in staging, and add Twilio Messaging support for Safety Pack alerts. | Staging can send a real OTP and one test Safety Pack SMS through Twilio; provider IDs/statuses are logged; no Twilio secret is available to browser code or any `VITE_` variable. | Gives the product one production SMS path while preserving the option to swap providers later. |
| 5 | P0 | Web invite phone verification can call Twilio behind a feature flag, but final invite submission still writes directly from the browser and can still submit `phone_verified=true`. | Replace browser invite writes with `submit-invite`, requiring an approved `phone_verifications` reference when phone verification is configured. | Visitor can submit an invite in the web app without Telegram only after real phone OTP succeeds; invalid/expired OTP fails; one active pending invite per verified phone per host is enforced; host receives exactly one pending invite. | Establishes trusted visitor identity and anti-duplicate behavior while preserving the web-first invite experience. |
| 6 | P0 | Telegram discovery phone verification is mocked with test code `123456`. | Reuse the provider-backed phone verification path from `telegram-webhook` for Telegram-origin invite attempts. Keep browsing available before phone verification, but gate the first invite link behind real OTP. | Visitor can browse profiles in Telegram before phone verification; when they tap Invite, the bot sends a real OTP, verifies it, and only then returns the selected `https://uinvite.me/<handle>?slot=<slot>` link. | Makes Telegram discovery production-safe without forcing Telegram before the web invite path. |
| 7 | P0 | Safety Pack activation only updates database status from the browser. There is no scheduled check-in, signed acknowledgement endpoint, escalation job, or real SMS alert. | Add `activate-safety-pack`, `ack-safety-pack`, `safety-checkin-reminder`, and `safety-alert` Edge Functions. Use Telegram for host check-ins and Twilio SMS for trusted-contact emergency/missed-check-in alerts. | Activating a pack schedules a check-in; All good marks completed; Emergency immediately alerts trusted contacts; missed check-in after grace period triggers one SMS escalation; every delivery is logged. | Turns the Safety Pack from UI into a real safety workflow. |
| 8 | P0 | Hosts need Telegram invite administration, but Telegram is not linked to host accounts and invite decisions are still primarily web UI actions. | Implement host Telegram linking, new-invite notifications, and inline Accept/Decline buttons backed by server-side `accept-invite` and `decline-invite`. | Linked host receives a Telegram message for a new invite, can accept or decline from Telegram, and web dashboard/date state updates consistently with no duplicate dates. | Gives hosts the intended admin channel and moves invite lifecycle to trusted backend logic. |
| 9 | P0 | Accepted-invite Telegram notification works for linked visitors, but host Telegram contact sharing is still minimal because host Telegram linking is not implemented. | Keep visitor post-submit Telegram opt-in, then add host Telegram linking and host-selected contact payloads for `telegram` or `instagram`. | Visitor can submit via web first, optionally link Telegram, and receive acceptance details containing exactly the host-selected contact method; host phone is never shared by default. | Adds accepted-invite notification without blocking web submission or forcing Telegram too early. |
| 10 | P1 | Turnstile CAPTCHA is required for public profile submits but is not implemented. | Add Turnstile to the public invite wizard for `/:handle` and verify the token server-side in `submit-invite`. | Public profile submissions without a valid Turnstile token are rejected server-side; token links can follow the chosen policy. | Reduces spam on public profiles. |
| 11 | P1 | Auto-decline rules and keyword moderation are configured in the UI/schema but not reliably enforced on submit. | Enforce enabled questions, auto-decline rules, note length, and keyword moderation in `submit-invite`. | Unsupported or missing required answers are rejected; configured auto-decline rules set status appropriately; flagged content shows moderation status to the host. | Keeps screening behavior consistent and prevents client-side bypass. |
| 12 | P1 | Safety link tokens are generated in the browser and acknowledgement links are not exposed as trusted server endpoints. | Generate signed or high-entropy safety tokens server-side, store only what is needed, and route acknowledgement URLs to a public Edge Function. | Tokens cannot be guessed, are scoped to one pack/action, and acknowledgement writes are idempotent. | Protects sensitive safety actions from spoofing. |
| 13 | P1 | Public schedule data is assembled by multiple browser queries and depends heavily on permissive RLS. | Add a `public-schedule` Edge Function or carefully audited RPC that returns only the public schedule payload needed by the UI. | Anonymous users can fetch valid public/link schedules but cannot enumerate private schedules, invitee data, or host private fields. | Reduces privacy risk and simplifies RLS. |
| 14 | P1 | Environment handling is not production-ready. `.env` is tracked and there is no `.env.example` documenting required variables. | Untrack real secrets, add `.env.example`, document frontend env vars and function secrets separately, and rotate exposed credentials if needed. | Fresh clone can be configured from `.env.example`; no secret-bearing `.env` file is committed; browser only receives publishable keys. | Prevents credential leakage and makes deployment repeatable. |
| 15 | P1 | SMS and notification delivery are not auditable or retryable yet. | Add `notification_outbox`, `notification_deliveries`, or `sms_messages` tables plus Twilio status callback handling and idempotency keys. | OTP, Telegram notifications, and Safety Pack SMS have provider IDs, statuses, timestamps, failure reasons, and safe retry behavior. | Makes notification failures observable instead of silent. |
| 16 | P2 | Supabase functions are only partially versioned in the repo; `telegram-webhook` and `accept-invite` exist, but the broader trusted backend is still not complete. | Keep every Edge Function in `supabase/functions`, add local tests, and deploy them from CLI/CI instead of dashboard-only edits. | All production backend code is committed beside migrations; functions can be tested locally and deployed repeatably from CLI/CI. | Makes the product independent from Lovable/dashboard-only state. |
| 17 | P2 | Local/staging verification is still mostly manual. | Add a lightweight smoke-test checklist or scripted E2E flow for local/staging: host signup, profile, schedule, invite, accept, Telegram opt-in, and Safety Pack. | A developer can run one documented command/checklist and verify the main product loop against staging Supabase. | Reduces deployment risk and makes future changes safer. |
| 18 | P2 | Demo pages exist but the sandbox behavior is not clearly isolated from production invite tables. | Keep demo submissions fully client-only, or route them to a dedicated demo table/function with purge policy. | Demo invite flow cannot create real invites, dates, SMS sends, or safety events. | Preserves trust and avoids noisy production data. |
| 19 | P2 | Telegram discovery now has an MVP loop, but production ranking and invite creation remain simple. | Improve discovery ranking/location and connect Telegram-origin invite handoff to the future trusted `submit-invite` flow. | Visitor can browse eligible profiles one by one, skip profiles, verify phone with real OTP when needed, and create the selected invite through the trusted backend path. | Hardens the discovery loop for production without replacing the richer web invite wizard. |
